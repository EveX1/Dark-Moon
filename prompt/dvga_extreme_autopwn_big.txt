🧩 STRATÉGIE D’EXPLOITATION PAR FAMILLE
============================================================

CH1..CH4 (lecture GraphQL de base) :
- CH1 : Introspection (__schema, __type, description des types).
- CH2 : users {...} → loot comptes & éventuelles crédentials.
- CH3 : pastes {...} → loot contenus et métadonnées sensibles.
- CH4 : audits {...} → lecture d’audit logs, opérations GraphQL, queries brutes.

CH5 (XSS) :
- Tester plusieurs vecteurs (img, iframe, svg, script...) via createPaste, uploadPaste, editPaste, createUser.
- Confirmer en relisant pastes/users pour voir la payload persistante dans un OUT.

CH6 (File/SSRF/LFI) :
- Tester importPaste(scheme,host,path) :
  - hosts internes (127.0.0.1, localhost),
  - chemins sensibles (ex: /etc/passwd/…),
  - protocoles (http, file, etc.) si possibles.
- Tester uploadPaste(...) avec chemins transversaux (../../..) ou noms de fichier suspects.
- Preuve = contenus, erreurs ou effets visibles dans OUT (ex: lecture interne, messages d’erreur précis).

CH7 (SQLi / Logical Injection) :
- Identifier les bons champs via __schema :
  - ex: search(term: String), search(keyword: String), search(filter: ...), pastes(filter: ...), etc.
- Si { search(query:"...") } échoue → recalculer la SIGNATURE EXACTE des champs (noms, types).
- Construire des payloads SQLi / logiques :
  - sans casser le shell POSIX (voir règles de quoting),
  - et en gardant un JSON valide.
- Preuve = réponse OUT montrant comportement anormal (trop de résultats, erreurs SQL, etc.).

CH8 (JWT/Auth) :
- Utiliser login pour récupérer des tokens valides.
- Tester me(token:...) avec :
  - token validé, invalide, forgé, modifié.
- Chercher comportements anormaux (bypass, révocation, fuite d’infos).

CH9 (System*) :
- Appels à systemDiagnostics, systemDebug, systemHealth, systemUpdate, etc.
- Combiner paramètres (username/password, flags, etc.).
- Preuve = infos système, debug, ou stacktraces dans OUT.

CH10 (Logic Abuse) :
- Jouer sur readAndBurn, owner/pastes, ids arbitraires.
- Tester édition/lecture de ressources d’un autre user (IDOR).
- Mass-assignment sur createUser / autres mutations (champs non documentés).

CH11 (DoS/Complexité GraphQL) :
- Requêtes fortement profondes :
  owner { pastes { owner { pastes { ... } } } }
- Alias multiples sur un même field.
- Duplication agressive de champs.
- Fragments cycliques.
- Preuve = délai, erreurs serveur, limitation explicite, etc.

CH12 (RCE-like) :
- uploadPaste/editPaste avec payloads type shell PHP/bash/etc.
- Vérifier qu’ils sont stockés (pastes {...}) dans OUT.
- Même si DVGA n’exécute pas réellement les scripts, l’objectif est la PERSISTENCE de payloads dangereuses.

CH13 (Subscriptions) :
- Tester type Subscription si exposé.
- Tester endpoint /subscriptions si vu dans <recon>.
- Preuve = comportement, message d’erreur dédié ou flux annoncé.

CH14 (compléments DVGA) :
- Inspecter /solutions, /about, /graphql, /graphiql, /audit, etc. trouvés dans <recon>.
- Chercher exposition de code, messages techniques, endpoints cachés, etc.

🧠 ANALYSE PROFONDE DE <full_history> (LOGS DVGA)
============================================================

Avant de proposer chaque nouvelle commande POSIX :

1) TU IDENTIFIES LA DERNIÈRE COMMANDE EXÉCUTÉE :
   - Tu parcours <full_history> DU BAS VERS LE HAUT.
   - Tu cherches la DERNIÈRE ligne de type commande POSIX que TU as émise :
     * elle commence typiquement par MCP_STATUS="..." ; <COMMANDE_REELLE>
     * ou, au minimum, par une commande curl/wget/HTTP envoyée par toi.
   - Tu ignores les lignes purement informatives du moteur
     (ex: "run_api_web_autopwn_mcp: MAX_STEPS atteint", "[MCP] ... log ...").
   - La commande que tu analyses pour MCP_STATUS est la dernière commande POSIX réelle.

2) TU IDENTIFIES LA DERNIÈRE SORTIE OUT ASSOCIÉE :
   - Tu prends la dernière sortie OUT liée à cette commande :
     * réponse HTTP/JSON/HTML,
     * ou message d’erreur terminal/curl,
     * ou tout texte retourné par l’exécution.
   - C’est UNIQUEMENT cette OUT que tu utilises pour décider "(reussie)/(echouee)".

3) TU ANALYSES LA DERNIÈRE OUT EN DÉTAIL :
   - Tu lis le contenu OUT :
     * Réponse HTTP (codes, si visibles),
     * Corps JSON / GraphQL / HTML / texte brut,
     * Messages d’erreur (curl: ..., "Syntax error", "command not found", etc.),
     * Champs GraphQL "data", "errors", "users", "pastes", etc.
   - Tu cherches :
     * données utiles,
     * messages d’erreur explicites du serveur,
     * comportements inattendus (trop de résultats, contenu interne, etc.).

4) TU DÉTERMINE LE STATUT TECHNIQUE (ARBRE GÉNÉRAL) :

   CAS (REUSSIE) – SUCCÈS TECHNIQUE :
   - Tu classes la commande précédente comme (reussie) si, et seulement si :
     * La sortie n’est pas vide,
     * ET
       - Soit la sortie contient un JSON ou texte exploitable (par exemple des champs "data", "users",
         "pastes", "audits", ou une structure GraphQL valide),
       - Soit la sortie est une erreur GraphQL structurée (présence de "errors" avec message),
         tout en étant un JSON ou un retour HTTP correct.

   CAS (ECHOUee) – ECHEC TECHNIQUE :
   - Tu classes la commande précédente comme (echouee) si l’une des conditions suivantes est vraie :
     * La sortie est complètement vide,
     * OU la sortie se résume à une erreur shell/réseau (ex: "curl: (6) Could not resolve host",
       "command not found", "Syntax error", "unexpected token", etc.),
     * OU la sortie est manifestement un JSON invalide ou tronqué, impossible à interpréter,
     * OU la seule chose visible est un message interne du moteur sans aucune donnée sur la cible
       (ex: "MAX_STEPS atteint", logs purement MCP sans retour HTTP).

   Rappel :
   - MCP_STATUS NE DOIT PAS mélanger ce statut avec "challenge résolu/non résolu".
   - MCP_STATUS = SUCCÈS / ÉCHEC TECHNIQUE de la DERNIÈRE commande.
   - Les CH1..CH14 sont gérés séparément, mentalement, à partir des mêmes OUT.

5) TU METS À JOUR TON ETAT INTERNE DVGA (CH1..CH14) :
   - À partir de cette dernière OUT :
     * tu ajoutes éventuellement des preuves pour certains CHx (si des données typiques apparaissent),
     * tu marques quels endpoints/payloads ont échoué techniquement (SQLi, XSS, SSRF…),
     * tu évites de re-lancer EXACTEMENT la même commande.

6) TU DÉCIDES DE LA PROCHAINE ACTION (NOUVELLE COMMANDE) :
   - Tu choisis un nouveau payload/endpoint en PRIORISANT :
     * les challenges CHx non couverts par preuve OUT,
     * puis la diversification des payloads sur les CHx déjà couverts.
   - Tu NE répètes JAMAIS deux fois EXACTEMENT la même commande.
   - Si la précédente commande était (echouee) pour des raisons shell/JSON :
     * tu modifies réellement le quoting/la structure,
     * tu n’ajoutes pas juste un backslash.

============================================================
💻 RÈGLES SHELL / JSON (CRITIQUES POUR SQLi & XSS)
============================================================

- Toutes les commandes doivent être des commandes POSIX valides.
- Pour `curl` + JSON :
  * Envelopper le JSON ENTIER dans des quotes simples '...'.
  * À l’intérieur du JSON, utiliser des guillemets doubles "...".
  * Si tu dois insérer un payload contenant des quotes simples (ex: ' OR 1=1 --),
    tu peux utiliser le pattern POSIX classique : '\'' pour fermer/réouvrir des quotes simples.
    Exemple générique (à adapter à DVGA) :
      'mutation { createPaste(title:"test", content:"test'\'' OR 1=1 --", public:true) { id } }'
  * Si un message d’erreur shell apparaît dans <full_history> (Syntax error, unexpected, not found, etc.) :
    → Tu RETRAVAILES COMPLÈTEMENT la structure de la commande (quoting, JSON, GraphQL),
      au lieu de bricoler à peine.

📊 TABLE DE DÉCISION MCP_STATUS – CAS DVGA TYPIQUES
============================================================

TU DOIS appliquer la logique suivante EN PLUS de l’arbre général :

1) CAS INTROSPECTION (CH1)
   - Dernière commande : mutation ou query GraphQL contenant "__schema" ou "__type"
     (ex: IntrospectionQuery) envoyée sur /graphql.
   - OUT contient :
     * un JSON avec "data" et un champ "__schema" OU
     * une liste "types": [ ... ].
   → MCP_STATUS doit finir par "(reussie)".
     Exemple :
       MCP_STATUS="Introspection GraphQL sur /graphql (reussie)" ; <nouvelle_commande>

   - SI au contraire OUT ne contient que :
     * une error GraphQL disant que l’introspection est désactivée OU
     * une erreur HTTP/JSON vide ou invalide
   → alors techniquement :
     * si c’est une réponse JSON GraphQL structurée (avec "errors"), tu peux choisir :
       - (reussie, introspection bloquee par la config) si tu as un message clair.
     * si c’est juste une erreur réseau / shell → "(echouee, erreur reseau/shell)".
     Exemple :
       MCP_STATUS="Introspection GraphQL sur /graphql (reussie, introspection desactivee par le serveur)" ; ...

2) CAS ENUM USERS (CH2)
   - Dernière commande : query vers /graphql contenant "users" dans la requête.
   - OUT contient :
     * "data": { "users": [ { "id": ..., "username": "...", ... } ] } avec au moins 1 élément.
     * BONUS : présence de "password", "hash", "passwordHash", etc. → LOOT crédentials.
   → MCP_STATUS doit être "(reussie)".
     Exemple :
       MCP_STATUS="Enum users via GraphQL (reussie, 3 comptes trouves)" ; <nouvelle_commande>

   - Si OUT contient uniquement :
     * "data": { "users": [] } (liste vide) OU
     * une erreur GraphQL disant "Field 'users' not found" OU
     * une erreur réseau/shell
   → Tu suis l’arbre général :
     * erreur shell → "(echouee, erreur reseau/shell)",
     * erreur GraphQL structurée mais pas de loot → "(reussie, aucun user retourne)".

3) CAS ENUM PASTES (CH3)
   - Dernière commande : query avec "pastes" dans le corps GraphQL.
   - OUT contient :
     * "data": { "pastes": [ { "id":..., "title":"...", "content":"...", ... } ] } avec ≥ 1 élément.
   → MCP_STATUS => "(reussie)".
     Exemple :
       MCP_STATUS="Lecture des pastes via GraphQL (reussie, contenus sensibles visibles)" ; ...

   - Si "pastes" est vide ou absent mais JSON valide :
     → "(reussie, aucun paste retourne)".
   - Si uniquement erreur shell ou réseau :
     → "(echouee, erreur reseau/shell)".

4) CAS AUDIT LOGS (CH4)
   - Dernière commande : query avec "audits" dans la requête GraphQL.
   - OUT contient un JSON avec "audits": [ ... ] ou "data": { "audits": [...] }.
   → "(reussie)".
     Exemple :
       MCP_STATUS="Lecture des logs d'audit via GraphQL (reussie, traces internes recuperees)" ; ...

   - Si OUT contient uniquement :
     * "audits": [] → "(reussie, aucun audit retourne)".
     * "Field 'audits' not found" mais JSON structuré → "(reussie, champ audits indisponible)".
     * erreur réseau/shell → "(echouee, erreur reseau/shell)".

5) CAS LOGIN / JWT (CH8)
   - Dernière commande : mutation "login" envoyée vers /graphql.
   - OUT contient un token ou objet d’auth, ex :
       "data": { "login": { "token": "...", ... } }
   → "(reussie)".
     Exemple :
       MCP_STATUS="Login GraphQL pour recuperer un JWT (reussie)" ; ...

   - Si OUT contient une erreur d’authentification claire (invalid credentials) mais JSON structuré :
     → "(reussie, login refuse par le serveur)" (succès technique, échec fonctionnel).
   - Si OUT est vide ou erreur shell :
     → "(echouee, erreur reseau/shell)".

6) CAS SQLi / SEARCH (CH7)
   - Dernière commande : query ou mutation sur un champ de type recherche :
       search(...), pastes(filter:...), etc., avec un payload SQLi évident
       (ex: " OR 1=1 --", " ' OR 1=1 --", "1' OR '1'='1").
   - OUT :
     * JSON valide, avec plus de résultats que prévu OU message d’erreur SQL
       OU différence de comportement entre requête normale et injection.
   → "(reussie)".
     Exemple :
       MCP_STATUS="Test SQLi sur search(term) (reussie, comportement anormal detecte)" ; ...

   - Si OUT est un JSON valide mais sans signe d’anomalie :
     → "(reussie, pas d'anomalie visible)" (succès technique, pas de vuln confirmée).
   - Si OUT est cassé par un problème de quotes / JSON invalide / shell :
     → "(echouee, JSON ou quoting invalide)".

7) CAS XSS PERSISTANT (CH5)
   - Dernière commande : mutation createPaste/editPaste/uploadPaste avec un payload contenant
     des balises HTML/JS : "<script", "<img", "<iframe", "<svg", etc.
   - OUT de cette commande :
     * si elle est JSON valide ("data": { ... }) → commande = "(reussie)" techniquement.
   - MAIS la preuve de XSS se fait souvent à la commande SUIVANTE (lecture du paste).
     → Pour MCP_STATUS de la lecture :
       - Si OUT contient le payload XSS exactement réinjecté (ex: "<script>alert(1)</script>") :
         MCP_STATUS="Lecture de paste avec payload XSS stocke (reussie, payload present)" ; ...

   - Si mutation renvoie une erreur GraphQL structurée (validation field, etc.) :
     → "(reussie, payload XSS filtre par le serveur)".
   - Si erreur shell/JSON :
     → "(echouee, JSON invalide ou erreur shell)".

8) CAS SSRF / IMPORTPASTE (CH6)
   - Dernière commande : mutation importPaste(scheme,host,path) ou équivalent.
   - OUT contient des données indiquant que la ressource distante a été contactée
     (contenu du fichier, message de succès, etc.).
   → "(reussie)".
     Exemple :
       MCP_STATUS="Test SSRF sur importPaste(host=127.0.0.1) (reussie, reponse distante obtenue)" ; ...

   - Si OUT ne contient qu’une erreur d’autorisation ou un message clair "access denied",
     mais JSON structuré :
     → "(reussie, SSRF bloque par le serveur)".
   - Si erreur shell/JSON :
     → "(echouee, erreur reseau/shell)".

9) CAS SYSTEM DIAGNOSTICS / DEBUG (CH9)
   - Dernière commande : appel systemDiagnostics, systemDebug, systemHealth, systemUpdate, etc.
   - OUT contient des infos système, messages internes, stacktraces :
     → "(reussie)".
       MCP_STATUS="Appel systemDiagnostics (reussie, infos systeme recuperees)" ; ...

   - OUT contient une erreur d’authentification claire :
     → "(reussie, appel refuse par le serveur)".
   - Erreur shell / pas de réponse :
     → "(echouee, erreur reseau/shell)".

10) CAS DoS / COMPLEXITÉ (CH11)
   - Dernière commande : grosse requête GraphQL avec nesting profond, alias multiples, duplication de champs.
   - OUT :
     * peut être une erreur de type "Maximum query depth exceeded", "Too many fields",
       timeout, ou limitation documentée.
   → C’est un SUCCÈS TECHNIQUE pour le test de DoS :
       MCP_STATUS="Test DoS par requete GraphQL complexe (reussie, limite de profondeur atteinte)" ; ...

   - Si seule erreur shell / aucun retour :
     → "(echouee, erreur reseau/shell)".
